## 0. 核心问题

Roundtable 要解决三个互斥的需求：

1. **AI 需要足够深的方法论**（对手类型、战意、风格交互等~1500 行知识）才能做出高质量预测
2. **但上下文窗口有限**，不可能每轮都塞全部知识
3. **多 AI 辩论时容易羊群效应**，保守/不自信的模型会跟风准确率高的模型

本设计通过 **分层注入 + 阶段切换 + 匿名辩论 + 加权投票** 同时解决这三个问题。

---

## 1. 整体架构：七层提示词注入

每个 AI 实例收到的是一个**运行时组装的 prompt**，不是固定文本。Python orchestrator 负责每次调用前拼好：

```
┌──────────────────────────────────────────┐
│  Layer 1: 身份与角色（版本化静态）        │ ← 迁移发布，每桌冻结版本
│  Layer 2: 核心方法论（压缩）              │ ← 从 skill 提取，变动极低频
│  Layer 3: 匿名同伴映射（每桌冻结）        │ ← 圆桌开始时分配；不向实例暴露自己的代号
│  Layer 4: 比赛上下文（每场冻结）          │ ← 从 MatchDataService 组装为输入快照
│  Layer 5: 历史战绩快照（每实例每桌冻结）  │ ← 开桌时从 DB 生成版本
│  Layer 6: 阶段指令（随阶段切换）          │ ← selection / score / bet / review 不同
│  Layer 7: 工具函数定义（静态）            │ ← 数据访问、计算等
│  ──────────────────────────────────────  │
│  最后：Phase-specific 输出格式规范        │ ← 本阶段要输出的结构化 Schema
└──────────────────────────────────────────┘
```

### 1.1 各层来源与维护

| 层 | 内容量 | 维护人 | 存储位置 |
|---|---|---|---|
| L1 身份角色 | ~50 行 | 开发 | `prompt_versions(key=identity)`；由迁移发布，后台只读 |
| L2 核心方法论 | ~350 行 | 管理员在后台·系统设置编辑 | `prompt_versions` 表，版本化；当前正式 key=`core_methodology` |
| L3 匿名代号 | 1 行 | 编排器运行时生成 | `roundtable_jobs.codename_seed` + 冻结映射；实例只看到其他人的代号 |
| L4 比赛上下文 | ~150–250 行 | Python DataService 组装 | `input_snapshots` + 来源记录关联 |
| L5 历史战绩 | ~30–50 行 | 开桌时从 DB 查询一次 | `history_context_versions` + `lesson_set_versions` |
| L6 阶段指令 | ~40–80 行/阶段 | 开发 + 管理员可调 | `prompt_versions` 表 |
| L7 工具函数 | ~60 行 | 开发 | `tools/` 目录 + 厂商中立 JSON Schema；适配器转换为各 Provider 格式 |

所有阶段合同以本文 §3.8 的 JSON Schema 为准。自然语言示例只解释语义，不得替代 Schema；无法通过 Schema 的响应按阶段失败处理，禁止用正则从自由文本中“抢救”投票。

### 1.2 消息角色与提示注入边界

“分层注入”不等于把所有文本拼成一个 system message：

- **可信指令层**：L1 身份、L2 核心方法论、L6 阶段指令、输出 Schema 与工具合同由平台版本库加载，映射到 Provider 支持的最高可信指令角色。
- **不可信数据层**：L4 比赛数据、L5 历史/lesson、匿名同伴安全上下文、管理员补充材料和工具返回值均作为类型化 data/tool message 传入；即使其中出现“忽略前文”“改用其他格式”等措辞，也只能视为数据，不能改变指令、工具权限或输出合同。
- 编排器对不可信文本执行 Unicode 规范化、长度/数组上限、控制字符与伪造角色标签过滤，并以 JSON/结构化字段序列化；不得把上游 HTML、Provider 原始响应或数据库自由文本直接插入可信指令模板。
- Provider 适配器必须保留上述角色边界；若某模型无法可靠区分指令与数据，必须在 G4 contract test 中判定该模型不支持相应阶段，禁止退化为无边界字符串拼接。

## 2. Layer 2: 核心方法论（已固化正式版本）

本节是从早期约 1500 行方法论材料中提炼并固化的运行时正式规则。早期材料只保留为设计来源，不是运行时依赖；**只保留判断型规则，删除操作型指令。**

### 2.1 存什么 vs 不存

| 保留（注入 prompt） | 删除（走工具/RAG） |
|---|---|
| 五条核心分析原则 | 数据源 URL 列表（走工具 `get_match_data`） |
| 对手类型分级 A/B/C 表格 | 搜索词模板（走工具内置） |
| 战意控分 7 类型表格 | XGBoost 公式（系统算，不是 AI 做的事） |
| 双边风格交互矩阵 5 行 | 体彩接口调用细节 |
| 淘汰赛专项规则（压缩） | 报告 HTML 渲染脚本 |
| 平局总进球二分规则 | 免责声明模板 |
| 赔率结构纪律（4 条） | 历史回测脚本 |
| 结论自洽性自检清单 | parlay-mode 详细配资算法 |
| 竞彩五大玩法口径约束 | 联赛球队 ID 映射表 |

### 2.2 压缩后的系统提示模板（L1+L2 合体）

```python
CORE_METHODOLOGY_VERSION = "core-methodology-v1.1"
CORE_METHODOLOGY_SYSTEM_PROMPT = """# 角色

你是 Alea 平台的一名专业足球比赛分析师。你的分析是多 AI 圆桌辩论的一部分：多个 AI 各自独立分析后匿名辩论、投票收敛。

# 核心分析原则

1. **不低估非传统强队的韧性**：亚洲、非洲、中北美球队在低位防守、身体对抗、定位球方面常有纪律优势。面对强队时不简单套「弱队模板」。
2. **也不低估超级巨星的破局能力**：超级巨星（Mbappé/Messi/Haaland/Kane 级别）在僵局时可凭个人能力撕开密防。面对此类球队时区分：超级巨星型 / 体系型 / 低效热门队。
3. **竞彩让球与亚盘必须区分**：竞彩让球是整数（如-1、+1），只出让胜/让平/让负三种结果，没有赢半输半。禁止把竞彩让球和亚盘混为一谈。
4. **市场热度不等于真实稳妥**：赔率下降不意味着该方向更稳。识别热门降水、让球不升盘、平赔下降等异常信号。
5. **输出必须包含风险**：避免绝对化表达（「稳赢」「必中」）。

# 对手类型分级（每场比赛前必判）

## A 崩盘型漏勺
- 近 1–2 场失 4+ 球，源于体系崩溃/速度打穿/士气崩溃
- → 强队进球级联放大，首选净胜上调一档

## B 纪律型铁桶
- 低位防守纪律严、近期失球少、靠体系不靠个人
- → 维持闷平/小胜剧本，净胜 ≤1 合理
- **铁桶降级**：若该铁桶近 1–2 轮刚逼平/零封同级别球队 → 强队直接胜置信度强制下调一档

## C 开放对攻型
- 攻击型阵型、高位逼抢、能进也能丢、零封率低
- → 总进球强制 +1 档，比分必含 BTTS（双方都进球）的比分，不得只押零封小胜
- ⚠️ 判 C 必须有近 1–2 场真实开放对攻的实然证据，不能拿纸面阵容或媒体预测凑

## U 暂不可判
- 近况、阵容或比赛阶段数据不足，无法可靠归入 A/B/C
- → 明确输出缺失项，降低方向置信度；不得为了满足分类格式编造依据

# 战意与控分分级（出比分前必判，与对手类型并列）

## 1. 生死·需大胜
- 积分靠后，必须赢且需净胜球才能出线 → 大分差显著上调，首选比分上移 1–2 档
## 2. 已出线·控分避强
- 已基本出线，为规避淘汰赛强敌而主动控分求平 → 平局概率显著上调，「平」提为独立主选项
## 3. 领头羊·求稳锁分
- 尚未数学出线但平局即基本锁定，为后续留力 → 压净胜，零封小胜 1:0 为首选
## 4. 只需平即达标
- 数学上平即出线/锁名次，赢无额外收益 → 把平(0:0/1:1)提为首选项，方向用「不败」
## 5. 双方皆生死·相互忌惮
- 两队都需要分，但都怕先丢球 → 低总进球、窄净胜、零封小胜与平局权重同时上调
## 6. 已出局·摆烂或放开打
- 数学出局 → 双向高方差，可能送分也可能爆冷，只给区间降置信
## 7. 正常争分
- 双方都还需要分、无控分动机 → 按常规实力/对手类型判断
## U. 战意暂不可判
- 积分规则、轮次、晋级形势或两回合信息不足 → 明确缺失项，不得强行归入 1–7；下调方向置信度

# 双边风格交互矩阵（出比分前必做）

不能只并列描述两队特点。必须合成一条「交互结论」— A 的进攻方式撞上 B 的防守与反击方式，会把比赛推向什么节奏？

| 交互类型 | 比赛会被推向 | 对比分影响 |
|---|---|---|
| 控球压上 vs 低位铁桶+长传反击 | 强队控球率高但高质量机会未必多；弱队拖慢比赛 | 净胜下调，1:0/1:1并列 |
| 边路爆点 vs 慢边卫 | 能持续制造高质量机会 | 净胜与总进球上调 |
| 高位开放 vs 高位开放 | 转换回合增多 | 总进球上调，必含 BTTS |
| 阵地慢热 vs 中路密集+门将强 | 低节奏消耗 | 0:0/1:1/1:0 权重上调 |
| 双方都务实/淘汰赛怕先丢球 | 上半场试探 | 半全场平平/平胜/平负，低总进球 |

# 淘汰赛专项规则（R32 起切换）

1. 战意单一化 — 每场生死，无控分避强场景
2. 竞彩按 90 分钟结算 — 强队晋级 ≠ 竞彩胜
3. 总进球偏低 — 取消末轮「+1 档」
4. 强队「直接胜」置信度低于 55% 时，比分层不得只押小胜，必须覆盖 90 分钟平

# 跨赛事阶段补充
1. 两回合淘汰赛必须读取 leg_number、首回合比分与当前 aggregate_score；领先方守和、落后方追分的动机按 90 分钟比分单独判断。
2. 中立场不得套用普通主场优势；venue_neutral=true 时取消未经数据支持的主场加成。
3. 联赛争冠、欧战资格、保级与无欲无求场景必须由积分差、剩余轮次和数学条件支持；无法取得条件时标记未知。

# 平局总进球高方差二分
凡判了「平/不输优先」，再问一句「这是死守平还是开放平？」：
- **死守低分平**：至少一方刻意龟缩 → 0:0/1:1
- **开放高分平**：双方都有攻击力、谁都不刻意死守 → 必含 2:2/3:3

# 赔率结构纪律
1. SP<1.30 的选项不得进入串关主腿（回报率不对称）
2. 方向层(胜平负)建议仓位优先
3. 比分投注 ≤ 总预算 10%
4. 淘汰赛正 EV 冷门票独立小额配资

# 竞彩口径硬约束（不可违反）
- 只使用五种玩法：胜平负、让球胜平负（整数）、比分、总进球数(0–7+)、半全场胜平负
- 禁止出现「大小球」「亚盘让球-0.5」「角球」等非竞彩玩法
- 结算按 90 分钟常规时间+伤停补时，不含加时和点球
- 比分含半场标注 `半场 x:x`

# 结论自洽性自检（输出前必过）
- 比分 ←→ 总进球：对齐（2:1 → 3 球档）
- BTTS ←→ 零封：判了 C 或对手 λ≥0.8 时禁止零封比分
- 胜负倾向 ←→ 竞彩让球：同向
- 半全场 ←→ 胜负：全场段一致
- 置信度 ←→ 措辞：低置信时避免堆砌推荐
- 风险栏 ←→ 推荐腿：写进风险的方向不得作为保底层单选
"""

# Layer 1 身份部分的迁移 seed；运行时按冻结版本从 prompt_versions 加载

你的分析不追求「准确率 100%」—足球比赛的随机性意味着不存在完美预测。你的目标是：基于可用数据，给出当前最合理的判断，并诚实标注不确定性。
```

### 3.3 Phase 2: 匿名辩论指令

```python
PHASE_DEBATE_INSTRUCTION = """
# 当前阶段：圆桌辩论（第{round_num}轮）

你正在与其他 AI 分析师进行圆桌讨论。每个人都在独立预测中提交了初步判断。

系统不会告诉你自己的匿名代号；其他参与者只能看到编排器分配给你的代号。你只需回应输入中出现的其他选手代号。

## 你可以做以下事情：

1. **坚守并解释**：如果其他 AI 的分析没有改变你的判断，可以重申你的观点，但必须有针对性地回应他们的论据（例如：「选手 C 提到[具体论据]，但根据[你的数据/推理]，我认为这个因素被高估了，理由是…」）

2. **改票并说明原因**：如果其他 AI 指出了你忽略的因素，你**应该**改变你的预测。改票是理性的，不是丢人的。格式：
   `改票：首选比分 1:1 → 2:1（半场 0:0 → 1:0）`
   `原因：[你被说服的具体证据/推理变化]`

3. **质疑/补充**：指出其他 AI 分析中可能的盲点。

## ⚠️ 重要规则：

① **不要因为「大多数人预测相同」就改票** — 每次改票必须基于具体论据，不是人数。共识不等于正确。

② **不要因为「某个 AI 看起来很自信」就跟风** — 自信不代表准确。审视数据分析，不是语气。

③ **不要攻击其他 AI 的「身份」** — 你不知道它们的厂商或模型，它们也不知道你的。只就论据辩论。

④ **每条回应尽量引用具体数据，但在数据不可用时诚实标注** — 「根据主场优势」太笼统，要写「根据主队近 10 场联赛主场胜率 70% 且客队客场场均失 1.8 球」。若数据注明「暂缺」，则基于已有信息做合理分析并标注「某数据缺失，结论基于有限信息」。

⑤ **如果你发现自己的数据与其他 AI 矛盾，先检查：** 双方是否引用同一冻结快照中的不同来源记录。若快照内无法消解，明确标注冲突和不确定性；不得刷新实时赔率或换用本桌之外的数据。

⑥ **保留改票的权利** — 可以因同一冻结快照中的已核验事实或更好的推理而调整判断。改票 + 记下具体依据比死扛更专业。
"""
```

`history_context_limits` 首版版本值为 `recent_match_limit=10`、`lesson_limit=5`，范围分别为 1–50 与 1–20。函数不设置本地默认值，调用方必须传入圆桌已冻结的设置版本，避免 PRD、数据库配置与 Prompt 组装出现三套默认值。

> **设计原则**：历史战绩不是用来给 AI「自我感觉好或差」的，而是让 AI 识别自己的系统性偏差。只有管理员已发布的复盘教训才能进入 active 集合；教训、最近赛果和校准数据在开桌时一起冻结。一个「偏乐观」的 AI 看到自己的方向置信度校准数据后，应该主动校正后续概率表达。

## 6. 辩论交互协议（核心设计）

### 6.1 匿名实现

```python
BASE_CODENAMES = [
    "Alpha", "Beta", "Gamma", "Delta", "Epsilon", "Zeta", "Eta",
    "Theta", "Iota", "Kappa", "Lambda", "Mu", "Nu", "Xi",
    "Omicron", "Pi", "Rho", "Sigma", "Tau", "Upsilon", "Phi",
]

def assign_codenames(
    instance_ids: list[str],
    codename_seed: str,
) -> dict[str, str]:
    """按冻结种子分配可复现代号；超过基础池时继续生成匿名序号。"""
    if len(instance_ids) != len(set(instance_ids)):
        raise ValueError("instance_ids must be unique")

    rng = random.Random(codename_seed)
    shuffled_ids = sorted(instance_ids)
    rng.shuffle(shuffled_ids)

    return {
        instance_id: (
            BASE_CODENAMES[index]
            if index < len(BASE_CODENAMES)
            else f"选手-{index + 1:02d}"
        )
        for index, instance_id in enumerate(shuffled_ids)
    }

def anonymize_verified_messages(
    messages: list[RoundtableMessage],
    codename_map: dict[str, str],
    own_instance_id: str,
    shuffle_seed: str,
) -> list[dict]:
    """只把完成事实核验的安全字段交给其他 AI，并去掉厂商/模型信息。"""
    result = []
    for msg in messages:
        if msg.instance_id == own_instance_id:
            continue
        safe_context = build_verified_peer_context(
            arguments=msg.arguments,
            fact_claims=msg.fact_claims,
            input_snapshot_id=msg.input_snapshot_id,
        )
        codename = codename_map[msg.instance_id]
        result.append({
            "speaker": codename,
            "position": safe_context.position,
            "arguments": safe_context.arguments,
            "verified_facts": safe_context.verified_facts,
        })
    rng = random.Random(shuffle_seed)
    rng.shuffle(result)
    return result
```

未经核验或与冻结快照不一致的事实主张不得进入 `safe_context`，也不得出现在最终共识理由中；可以保留“存在一条未核验主张”的审计状态，但不能把其原文继续传播给其他实例。

**关键点**：匿名是双向的——AI 不知道自己的代号，也不知道其他 AI 的真实身份。代号只用于其他参与者的安全上下文和用户端审计展示，完整映射仅在 orchestration 层保存。

### 6.2 辩论流程

```
  Phase 0A: 独立选场提名
    每个 AI 阅读冻结候选池摘要并独立提名
       │
       ▼
  Phase 0B: 选场匿名辩论（固定 1 轮，可配置为 0–2 轮）
       │
       ▼
  Phase 0C: 选场终投
    形成当桌比赛集合；若 0 场入选，生成「今日休息」公告草稿并结束
       │
       ▼
  对每场入选比赛分别创建 child run，执行：

  Phase 1: 独立比分预测
    ┌─────┐ ┌─────┐ ┌─────┐
    │ AI-1│ │ AI-2│ │ AI-3│ ← 各自独立，不交流
    └──┬──┘ └──┬──┘ └──┬──┘
       │       │       │
       ▼       ▼       ▼
   预测写入 roundtable_events（Supabase PG）
       │
       ▼
  Phase 2: 比分辩论第 1 轮
    AI-1 看到: [Alpha 的预测+推理] [Beta 的预测+推理]
    AI-2 看到: [Alpha 的预测+推理] [Gamma 的预测+推理]
    AI-3 看到: [Beta 的预测+推理] [Gamma 的预测+推理]
    ↑ 每人都看不到自己对应的代号，看到其他人匿名。顺序随机打乱。
       │
       ▼
  Phase 2b: 比分辩论第 2 轮（可选）
    同上，带上第 1 轮的全部匿名发言
       │
       ▼
  Phase 3: 比分终投
    AI-1 投票 | AI-2 投票 | AI-3 投票
       │
       ▼
    Orchestrator 分别计算每场加权投票结果
   （权重 = 由综合分计算并经厂商家族归一化的 `w_i`，圆桌开始时已冻结）
       ▼
  等待所有 child run 进入终态
    单场失败/无 quorum 只排除该场，不抹掉其他成功场次
       │
       ▼
  Phase 4A: 独立组单提案
    各 AI 基于有效终投结论 + 冻结可售选项提出单一方案或 no_bet
       │
       ▼
  Phase 4B: 组单匿名辩论
       │
       ▼
  Phase 4C: 组单终投
    输出获选方案；合法结果允许 no_bet（零注、零金额、零风险占用）
    no_bet 与 bet 最终仍并列时按保守规则采用 no_bet，bet 候选仅留审计
```

### 6.3 防羊群效应机制（关键）

| 机制 | 设计 | 为何有效 |
|---|---|---|
| **Phase 1 独立** | 辩论前各自已经做出预测，写入 DB | 即使辩论中改票，原始判断也入了审计。改票只能新增理由，不能覆盖初稿 |
| **匿名代号** | 不知道对面是「Claude」还是「GPT」，只看论据 | 消除「品牌偏信」— 不会因为「Claude 说的所以更可信」 |
| **打乱顺序** | 匿名发言随机排序，不固定发言顺序 | 防止「第一个发言的锚定效应」 |
| **辩论指令第①条** | 显式禁止「因为大多数人相同就改票」 | 从 prompt 层面给 AI 独立意志 |
| **辩论指令第③条** | 显式禁止判断其他 AI 的模型身份 | 只能就论据辩论 |
| **加权投票** | 终投按由历史综合分计算并经厂商家族归一化的 `w_i` 加权，不是简单多数 | 历史表现与厂商家族上限会改变各票影响力，增加同厂商实例不会线性放大厂商票权 |
| **视觉金边** | 前端展示时高准确率 AI 的头像有金边 | 用户也能识别「这个 AI 历史更准」，不受票数迷惑 |
| **熔断按钮** | 管理员可随时「跳过辩论直接终投」 | 如果辩论陷入死循环或无用争吵，管理员可中断 |

## 7. 对抗「菜而不自知」：上下文注入策略

PRD 要求「让对应 AI 知道自己的历史预测信息、预测准确率等信息，免得某些 AI 预测时，菜而不自知」。

实现方式：在每一轮调用之前，组装带角色边界的 `PromptEnvelope`：

```python
def build_instance_context(
    instance: AIInstance,
    job_snapshot: PromptJobSnapshot,
    phase: str,
    *,
    target_match_id: str | None = None,
    peer_context: list[dict] | None = None,
) -> PromptEnvelope:
    trusted_instructions = []
    data_messages = []

    trusted_instructions.append(
        load_prompt_version(job_snapshot.identity_prompt_version)
    )
    # 不注入 own codename；模型只在辩论阶段看到其他人的匿名代号。
    if phase != "review_methodology":
        trusted_instructions.append(load_prompt_version(
            job_snapshot.core_methodology_version
        ))

    if phase.startswith(("select_", "score_", "bet_")):
        data_messages.append(serialize_data_message(
            kind="frozen_history",
            value=load_history_context(
                job_snapshot.history_context_version_ids[instance.id],
                db,
            ),
        ))

    if phase.startswith("select_"):
        data_messages.append(serialize_data_message(
            kind="selection_scope",
            value=build_selection_context(
                job_snapshot.selection_scope_snapshot_id
            ),
        ))
    elif phase.startswith("score_"):
        if not target_match_id:
            raise ValueError("score phase requires target_match_id")
        data_messages.append(serialize_data_message(
            kind="match_input_snapshot",
            value=load_input_snapshot_context(
                job_snapshot.match_input_snapshot_ids[target_match_id]
            ),
        ))
    elif phase.startswith("bet_"):
        data_messages.append(serialize_data_message(
            kind="bet_context_snapshot",
            value=build_bet_context(
                job_snapshot.bet_context_snapshot_id
            ),
        ))
    elif phase == "review_prediction":
        data_messages.append(serialize_data_message(
            kind="postmatch_review_context",
            value=load_postmatch_review_context(
                job_snapshot.postmatch_review_context_snapshot_id,
                instance_id=instance.id,
            ),
        ))
    elif phase == "review_methodology":
        data_messages.append(serialize_data_message(
            kind="methodology_review_context",
            value=load_methodology_review_context(
                job_snapshot.methodology_review_context_snapshot_id
            ),
        ))
    else:
        raise ValueError(f"unsupported phase: {phase}")

    if peer_context:
        data_messages.append(serialize_data_message(
            kind="verified_anonymous_peer_context",
            value=peer_context,
        ))

    trusted_instructions.append(load_phase_instruction(
        job_snapshot.phase_prompt_versions[phase]
    ))

    return PromptEnvelope(
        trusted_instructions=trusted_instructions,
        data_messages=data_messages,
        tools=load_tool_contract(
            job_snapshot.tool_contract_versions[phase]
        ),
        response_schema=load_output_schema(
            job_snapshot.output_schema_versions[phase]
        ),
    )
```

## 10. 总结：应对你的两个担忧

### 担忧 A：「有些模型会陷入死循环，在那钻牛角尖」

**防护层：**

| 防护 | 实现 |
|---|---|
| 固定轮数 | 默认只有 1 轮辩论，最多 2 轮，没有无限制自由讨论 |
| 熔断按钮 | 管理员随时可「跳过辩论直接终投」— 辩论再烂也不卡死 |
| Phase 1 独立预测已入账 | 即使辩论炸了，每个 AI 的初稿已经存在，可以直接跳过辩论用初稿投票 |
| Celery 任务超时 | 每个 AI 单次调用有最大等待时间（如 60s），超过标记「缺席」 |

### 担忧 B：「有些保守的模型给出的理由会影响其他模型的判断」

**防护层（同 §6.3）：**

| 防护 | 如何生效 |
|---|---|
| 匿名 | 保守模型如果来自「Claude」— 对面不知道它是 Claude，只看论据 |
| 辩论指令第①条 | 显式禁止「因为大多数人相同就改票」— 从 prompt 层面削弱跟风 |
| 加权投票 | 保守但准确率低的模型即使多，加权后权重也不一定高 |
| 改票需附理由 | 改票不是简单的「我改 2:1」— 必须写「改票原因：[具体论据]」，入审计后可追溯 |
| 高准确率金边 | 用户能直观看到「这个 AI 以前很准，这个不准」，不会被票数迷惑 |

```python
def check_methodology_proposal_conditions(db: Database):
    """
    后台定时任务：扫描 lessons 表，看是否有同类模式达到阈值。
    """
    settings = load_versioned_methodology_trigger_settings(db)
    aggregate_patterns = db.query("""
        SELECT category, normalized_pattern_hash,
               COUNT(DISTINCT review_id) AS review_count,
               COUNT(*) AS total_lessons,
               array_agg(DISTINCT ai_instance_id) AS involved_ais
        FROM lessons
        JOIN reviews ON reviews.id = lessons.review_id
        WHERE reviews.status = 'published'
          AND lessons.status = 'active'
          AND NOT methodology_proposal_generated
          AND (
              :lookback_days IS NULL
              OR lessons.published_at >= NOW() - make_interval(days => :lookback_days)
          )
        GROUP BY category, normalized_pattern_hash
        HAVING COUNT(DISTINCT review_id) >= :distinct_match_threshold
            OR COUNT(*) >= :lesson_count_threshold
    """, params=settings)

    # 返回与 aggregate_patterns 相同的规范化字段合同
    streak_patterns = find_consecutive_error_patterns(
        db,
        published_only=True,
        active_only=True,
        min_consecutive_matches=settings.consecutive_error_threshold,
        lookback_days=settings.lookback_days,
    )
    patterns = deduplicate_patterns(
        aggregate_patterns + streak_patterns
    )

    for pattern in patterns:
        proposal = MethodologyProposal(
            title=f"方法论调整：{pattern.category}",
            evidence_summary=(
                f"在 {pattern.review_count} 场复盘中出现 "
                f"涉及 {len(pattern.involved_ais)} 个 AI"
            ),
            status="pending_review",
        )
        db.insert(proposal)
        # 标记这些 lessons 已关联提议，避免重复生成
```

**提议触发阈值（均来自版本化配置）：**
- 单个模式出现在 ≥3 场不同比赛的复盘中
- 或 同一规范化模式累计 ≥5 条相关 lessons
- 或 某 AI 连续 5 场以上在同一类别犯错
- 证据回看窗口默认 `null`（全部有效历史）；管理员可改为最近 N 天，修改只影响后续扫描，不改变既有提议的证据集合

```python
    回测是评审圆桌的前置条件 — 没有回测数据，AI 无法评估修改效果。
    """
    config = load_versioned_backtest_execution_config(db)
    if config.attempts_per_instance < 2:
        raise InvalidBacktestConfig(
            "attempts_per_instance 必须 ≥2；Gate 验证后发布的配置值才可用于正式评审"
        )

    # 固定样本、实例、连接、模型、参数、Schema、工具快照、评价器和随机种子集合。
    samples = select_backtest_samples(
        criteria=proposal.category,
        limit=max(20, config.sample_size),
        required_instance_versions=config.instance_versions,
        require_complete_frozen_context=True,
    )
    if len(samples) < 20:
        raise InsufficientSampleError(
            f"回测样本不足：仅 {len(samples)} 场（要求 ≥20 场）"
        )

    backtest_snapshot = freeze_backtest_snapshot(
        proposal_id=proposal.id,
        sample_ids=[sample.id for sample in samples],
        sample_context_refs={
            sample.id: {
                "input_snapshot_id": sample.input_snapshot_id,
                "history_context_version_ids": (
                    sample.history_context_version_ids
                ),
                "lesson_set_version_ids": sample.lesson_set_version_ids,
            }
            for sample in samples
        },
        instance_versions=config.instance_versions,
        attempts_per_instance=config.attempts_per_instance,
        generation_parameter_versions=config.generation_parameter_versions,
        output_schema_version=config.output_schema_version,
        tool_contract_version=config.tool_contract_version,
        evaluator_version=config.evaluator_version,
        random_seeds=config.random_seeds,
    )

    prediction_records = []
    for sample in samples:
        # 预测执行进程只有赛前快照权限，不能读取 result_version_id、
        # 赛后事件、复盘或 lesson。
        input_snapshot = load_input_snapshot(sample.input_snapshot_id)
        for variant in ["OLD", "NEW"]:
            methodology_version = (
                proposal.base_prompt_version
                if variant == "OLD"
                else proposal.candidate_prompt_version
            )
            for instance_version in backtest_snapshot.instance_versions:
                history_context = load_history_context(
                    sample.history_context_version_ids[instance_version.id],
                    db,
                )
                for attempt_index in range(
                    backtest_snapshot.attempts_per_instance
                ):
                    prediction_records.append(
                        await run_schema_validated_prediction(
                            instance_version=instance_version,
                            methodology_version=methodology_version,
                            input_snapshot=input_snapshot,
                            history_context=history_context,
                            lesson_set_version_id=(
                                sample.lesson_set_version_ids[
                                    instance_version.id
                                ]
                            ),
                            generation_parameter_version=(
                                backtest_snapshot.generation_parameter_versions[
                                    instance_version.id
                                ]
                            ),
                            output_schema_version=(
                                backtest_snapshot.output_schema_version
                            ),
                            tool_contract_version=(
                                backtest_snapshot.tool_contract_version
                            ),
                            seed=(
                                backtest_snapshot.random_seeds[attempt_index]
                                if instance_version.capabilities.supports_seed
                                else None
                            ),
                            variant=variant,
                        )
                    )

    # 只有 OLD/NEW 的所有预测完成并锁定后，独立评价进程才加载赛果。
    result_versions = load_confirmed_results_for_evaluator(
        [sample.result_version_id for sample in samples],
        evaluator_version=backtest_snapshot.evaluator_version,
    )
    metrics = evaluate_paired_predictions(
        predictions=prediction_records,
        results=result_versions,
        dimensions=[
            "exact_score",
            "had_direction",
            "total_goals",
            "half_full",
            "invalid_output_rate",
            "execution_failure_rate",
        ],
        unit_of_analysis="match",
        confidence_interval="paired_bootstrap_95",
    )

    return BacktestResult(
        snapshot_id=backtest_snapshot.id,
        total_matches=len(samples),
        attempts_per_instance=backtest_snapshot.attempts_per_instance,
        metrics=metrics,
        paired_deltas=metrics.paired_deltas,
        confidence_intervals=metrics.confidence_intervals,
    )
```

同一套样本、实例、模型连接、当时的历史上下文/教训集、参数、Schema、工具快照与尝试计划必须同时用于 OLD/NEW；Provider 支持 seed 时使用同一可控随机种子集合，不支持时明确记录为 `null` 并依靠重复尝试估计波动。不得只展示一个无类型的总准确率。评审页面分别展示四类预测指标、无效输出率、执行失败率、配对差值与区间估计。评审圆桌中，每个 AI **必须基于回测数据做判断**，禁止仅凭直觉投票。

#### 管理员确认

终投结果：

- **支持（加权票 ≥ 60%）** → 管理员看到二次确认弹窗：「写入方法论的变更将影响此后所有圆桌的 L2 系统提示。确认发布？」

- **反对（加权票 ≥ 60%）** → 提议标记「已驳回（AI 评审未通过）」，归档

- **其余结果**（含「修改后再审」得票最高，或支持/反对均未达 60%）→ 进入「修改后再审」，当前方法论保持不变；提示管理员补充材料或调整提议文本，保存为新修订后重新回测、重新发起

#### + 管理员绕过权

管理员有权「跳过 AI 讨论直接修改方法论」— 如果人类发现一个显性问题，不需要等 AI 同意。该绕过操作**必须填写理由（禁止留空）并写入审计记录（记录操作者、时间、理由、修改前后内容对比）**。

### 11.6 对「自毁风险」的四重保险

| 保险 | 实现 |
|---|---|
| **回测强制** | 没有回测数据，AI 评审圆桌无法启动。回测 ≥20 场 |
| **投票不通过即归档** | 反对票 ≥60% 时提议关闭，不会反复弹出 |
| **管理员最终确认** | AI 说「支持修改」不等于改了 — 必须管理员点确认键 |
| **版本回溯** | 任何版本可一键回滚到上一版 |

### 11.7 「中途可调整」实现

管理员在方法论评审圆桌直播中做两件事：

1. **发表补充观点** → 作为明确标识的「管理员备注」注入到 AI 的下一条消息中；观点可直接讨论，事实性陈述仍须绑定来源并通过核验
2. **形成下一修订** → 管理员可以选中某条评审结论，点「采纳为提议修改方向」；一旦保存正文修改，当前 run 立即结束，新修订重新执行 OLD/NEW 回测后才能另起评审

### 11.8 需求意图与实现映射

| 需求意图 | 实现方式 |
|---|---|
| 复盘后发现需要调整方法论 → 按钮入口 | 系统自动聚合同类 lesson 达到阈值才生成提议；管理员在设置页看到提议列表，点「讨论此提议」 |
| 多个 AI 一块判断是否合理 | 方法论评审圆桌，AI 用评审专用 prompt，不是预测 prompt |
| 其他 AI 说出自己的见解 | 匿名辩论，管理员围观 |
| 中途可调整 | 管理员可在当前 run 发补充观点；采纳评审结论修改正文时生成新修订，并强制重新回测、另起评审 |
| 管理员二次确认后才修改 | 终投后管理员确认弹窗 |
| 方法论注入设置页面 | L2 prompt 以版本化配置显示在「后台·系统设置」，可直接手动编辑 |
| 平台将 AI 调整结果导入配置 | 管理员确认后自动写入 prompt_versions 表，自增版本号 |
